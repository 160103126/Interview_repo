## 🟢 Simple (Fundamentals)

### Q1: What is a Vector Database and why not just use PostgreSQL?

**Answer:**

A Vector Database is purpose-built to store, index, and query high-dimensional mathematical vectors (embeddings) generated by ML models.

**Why not PostgreSQL?**
Traditional relational databases (PostgreSQL, MySQL) are optimized for exact-match queries (`WHERE id = 5` or `name LIKE '%John%'`). 
Vector databases are optimized for **Approximate Nearest Neighbor (ANN)** search. When you query a vector DB, you are asking: *"Find the 5 vectors in this 1-billion-vector database that are mathematically closest to my query vector."*
Traditional DBs would have to do a full table scan, calculate the distance between the query vector and *every single row*, and sort the results. This is O(N) and takes minutes. Vector DBs use specialized indexes to do this in milliseconds.

*(Note: PostgreSQL CAN do this using the `pgvector` extension, which adds specialized ANN indexing).*

---

### Q2: What are the common Distance/Similarity Metrics?

**Answer:**

How do we define "closest" when comparing two vectors?

1. **Cosine Similarity:** Measures the *angle* between two vectors. It completely ignores the magnitude (length) of the vectors.
   - Range: -1 (opposite) to 1 (identical).
   - *Best for:* NLP text embeddings (like OpenAI's `text-embedding-3`). We only care about the semantic direction of the word, not how frequent it is.
2. **Dot Product:** Measures both angle and magnitude. 
   - *Note:* If your vectors are perfectly normalized (length = 1), Dot Product and Cosine Similarity are mathematically identical, but Dot Product is faster to compute on a GPU.
3. **Euclidean Distance (L2):** Measures the straight-line distance between two points in space. 
   - *Best for:* Computer vision, facial recognition.

## 🟡 Medium (Intermediate)

### Q3: How do Approximate Nearest Neighbor (ANN) indexes work?

**Answer:**

If we can't calculate the distance to every vector (O(N)), we must use an approximation algorithm. We sacrifice 1% of perfect accuracy for a 1000x speedup.

**Common ANN Techniques:**
1. **Tree-based (Annoy):** Splits the high-dimensional space into a binary tree using random hyperplanes. To search, you just traverse down the tree. (Fast, but accuracy drops in high dimensions).
2. **Quantization (Product Quantization - PQ):** Compresses vectors. It splits a 1536-dim vector into 8 sub-vectors, and clusters them. It stores short IDs instead of massive float arrays. Reduces memory footprint massively but loses precision.
3. **Graph-based (HNSW):** The gold standard. (See next question).

---

### Q4: Explain HNSW (Hierarchical Navigable Small World).

**Answer:**

HNSW is the underlying index algorithm used by almost every modern vector database (Pinecone, Qdrant, Milvus, pgvector).

**How it works:**
It is inspired by the "Six Degrees of Kevin Bacon" phenomenon and Skip Lists.
1. It builds a multi-layered graph of all vectors.
2. The **bottom layer** contains *all* vectors heavily connected to their closest neighbors.
3. The **top layer** contains only a few random "hub" vectors with very long connections.
4. **The Search:**
   - You enter at the top layer. You greedily jump to the hub vector closest to your query.
   - You drop down a layer. You search the local neighborhood of that hub.
   - You repeat until you hit the bottom layer, zooming in exponentially fast on the true nearest neighbors.
   
*Performance:* O(log N) search time. *Drawback:* Very high memory footprint (maintaining the graph connections).

## 🔴 Hard (Advanced / MAANG-level)

### Q5: How does Metadata Filtering work in a Vector DB? (Pre-filtering vs Post-filtering)

**Answer:**

A common query is: *"Find chunks similar to 'Remote Work' WHERE department='Engineering'"*.

**1. Post-filtering (Bad):**
- The DB runs a pure vector search and finds the Top-10 closest chunks.
- Then, it applies the SQL-like filter. If 9 of those chunks belong to 'HR', they are discarded.
- *Result:* You only return 1 chunk to the user, even though they asked for 10. The recall is terrible.

**2. Pre-filtering (Hard to implement):**
- The DB applies the SQL filter first, isolating only 'Engineering' chunks.
- Then, it runs the vector search on that subset.
- *Problem:* If 'Engineering' represents only 0.1% of the database, the HNSW graph is effectively destroyed. The "nodes" you need to jump to are missing, breaking the search path.

**3. Single-Stage / Custom Indexing (Modern standard):**
Databases like Pinecone and Qdrant use custom algorithms that evaluate metadata filters *simultaneously* during the HNSW graph traversal, ensuring strict K-results while maintaining high speed and perfect recall.

### Q6: Compare popular Vector Databases (Pinecone vs Milvus vs pgvector).

**Answer:**

| Feature | Pinecone | Milvus | pgvector (PostgreSQL) |
|---------|----------|--------|-----------------------|
| **Deployment** | Managed SaaS (Serverless) | Open-source, Cloud-native | Postgres Extension |
| **Architecture** | Proprietary, fully managed | Highly distributed, microservices | Monolithic (Standard DB) |
| **Best For** | Fast prototyping, no devops | Massive scale (billions of vectors) | Adding vector search to existing relational DBs |
| **Drawbacks** | Closed source, expensive at scale | Complex deployment (needs Kubernetes/Docker) | HNSW graph can bloat the DB, scaling reads is harder |
